# 现代 AI API 聚合系统架构设计文档 (基于 Rust)

## 1. 系统概述与设计目标

本系统旨在构建一个高可用、高性能的大语言模型（LLM）及其他 AI 能力的 API 聚合网关。系统对外屏蔽底层不同供应商（OpenAI, Claude, Azure, 本地模型等）的差异，对内实现负载均衡、密钥池管理、弹性容错与精确计费。

**核心设计目标：**
*   **极致性能与零延迟毛刺**：利用 Rust 的无 GC 特性，确保高并发流式（SSE）输出时的极低延迟与平滑吞吐。
*   **无缝透传与容错**：对客户端保持 100% 的 OpenAI 兼容性；上游渠道故障时实现无感知的自动重试。
*   **高精度实时计费**：支持基于 Token 的流式边算边扣，确保平台零坏账。

## 3. 核心技术栈选型

| 组件分类        | 技术选型                 | 选型理由                                                     |
| :-------------- | :----------------------- | :----------------------------------------------------------- |
| **开发语言**    | **Rust (Edition 2021)**  | 零 GC 毛刺，极致内存安全，超高并发处理能力。                 |
| **异步运行时**  | **Tokio**                | Rust 异步生态事实标准，极其强大的流处理（Stream）和任务调度。 |
| **Web 框架**    | **Axum**                 | 基于 Tower 中间件，类型安全，极其适合编写复杂的请求 Pipeline。 |

## 4. 核心模块详细设计

### 4.1 核心数据面：Rust 异步网关

这是整个系统的核心，负责处理实际的 LLM 流量。

#### A. 请求与响应流水线
采用 Axum 中间件机制，构建处理流水线：
1.  **解析拦截**：验证客户端 `Authorization`。
2.  **额度预扣减**：通过 Redis Lua 脚本，根据预估的 Max Tokens 原子扣减用户配额。若余额不足则直接返回 429。
3.  **路由选择**：根据请求的 `model` 参数，寻找可用渠道池。过滤掉被标记为“熔断中”的渠道，根据优先级和权重选择一个。
4.  **请求映射**：将客户端标准请求转换为上游渠道所需格式（如替换 Header、重写 Claude 到 OpenAI 的差异参数）。

#### B. 流式无缝透传与计费
*   **痛点**：大模型输出是 SSE（Server-Sent Events），如果全量缓存再返回，用户会面临长时间白屏。
*   **Rust 方案**：使用 `reqwest` 获取上游响应的 `Stream`。通过 Axum 的 `Body::from_stream`，将上游的字节流**直接且零拷贝**地透传给客户端。
*   **流中计费**：在透传的同时，内部启动一个异步 Task，实时拆解 SSE 数据包（`data: {...}`），提取新增的 `content`，使用 `tiktoken-rs` 进行流式 Token 计数。
*   **最终对账**：流结束后，根据实际消耗的 Token 计算真实费用，回退预扣减的多余额度，通过 NATS 异步发送日志入库。

### 4.2 渠道健康与容错路由

网关必须具备“抗灾害”能力，避免上游波动影响终端用户。

*   **优先级与权重**：支持配置多级渠道。优先打到 P0 渠道，若 P0 不可用，降级到 P1。
*   **自动重试**：
    *   如果上游返回 `429 Too Many Requests`、`500 Internal Server Error` 或**请求超时**。
    *   Rust 网关会自动**切断**当前流，将请求重新交给路由器，尝试下一个健康渠道（同一请求最多重试 N 次）。
*   **熔断机制**：
    *   如果某个渠道连续 N 次请求均失败（如鉴权失败、余额耗尽），网关会在 Redis 中将其标记为 `Disabled`，并设置 TTL（如 5 分钟）。
    *   在 TTL 到期前，路由器将直接忽略该渠道。

### 4.3 业务控制面：后台管理系统

提供友好的 Web UI，主要操作数据库和 Redis，不处理大模型流量。
*   **用户系统**：令牌生成、额度充值、兑换码发放。
*   **渠道管理**：配置上游 API Key、测试连通性、动态调整权重。
*   **数据看板**：基于 Prometheus 暴露 Metrics（每分钟请求数 QPM、平均延迟 P99、不同模型消耗趋势），使用 Grafana 展示。

## 5. 关键技术难点与 Rust 应对方案

#### 难点 1：流式 SSE 的解析与改写 (格式转换)
**场景**：对外暴露 OpenAI 格式，但实际上游调用的是 Claude 或文心一言。
**Rust 解决方案**：
不依赖第三方 SDK，手写流式拦截器。利用 Rust 的枚举和模式匹配，逐块读取 `reqwest::Bytes`，将其反序列化为目标模型的 SSE 结构体，映射好字段（如将 `delta.content` 映射为 `choices[0].delta.content`），重新序列化为标准的 OpenAI SSE 格式再向下发送。

#### 难点 2：高并发下的防额度透支
**场景**：用户余额仅剩 10，但同时发起 10 个并发请求，如果不加锁，都会判定余额充足，最终导致用户余额变成 -100。
**Rust 解决方案**：
完全依赖 Redis 服务端的单线程特性和 **Lua 脚本**。
```lua
-- 伪代码：Redis 原子预扣减 Lua 脚本
local current_balance = redis.call("GET", KEYS[1])
if tonumber(current_balance) >= tonumber(ARGV[1]) then
    redis.call("DECRBY", KEYS[1], ARGV[1])
    return 1 -- 扣减成功
else
    return 0 -- 余额不足
end
```
Rust 端只需通过高性能连接池发送该脚本，确保分布式环境下的绝对一致性。

#### 难点 3：异步长耗时任务
**场景**：Midjourney 画图、视频生成等 API 耗时可能长达数分钟，会阻塞 Rust 的线程池。
**Rust 解决方案**：
引入独立的后台 Task 管理器（基于 Tokio 的 `spawn` 或 NATS 队列消费）。
1. 网关接收到画图请求，向上游提交，获取 Task ID。
2. 将 Task ID 存入数据库并返回给客户端。
3. 后台启动一个独立的 Rust Task 定时轮询上游状态，完成后通过 Webhook 推送给客户端。

## 6. 部署与可观测性设计

*   **容器化部署**：使用多阶段构建 编译 Rust。第一阶段使用 `rust:latest` 编译生成极小的二进制文件，第二阶段将其拷贝入 `debian:bookworm-slim`，最终镜像大小可控制在 50MB 以内。
*   **无状态扩容**：网关本身不存储任何状态，所有的鉴权、配额、渠道健康状态均存储在 Redis 中。因此可以通过 K8s 或 Docker Compose 随意横向扩容 Rust 节点。
*   **分布式追踪**：在请求入口生成 `X-Request-ID`，注入到日志和响应头中。结合 OpenTelemetry SDK for Rust，可以将一次请求在“网关拦截 -> 渠道路由 -> 等待上游响应 -> 计费”各个阶段的耗时打点上报，方便定位性能瓶颈。
