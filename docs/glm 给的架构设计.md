参考 “New API” (以及其前身 One API) 的设计思想，一个现代化的 **API 聚合系统**（主要针对大语言模型 LLM 及其他 AI 能力）的核心目标是：**屏蔽底层不同供应商的差异，提供统一的对外接口，实现负载均衡、计费、鉴权和渠道管理。**

以下为您设计一个高可用、高扩展的 API 聚合系统架构方案：

### 二、 核心模块详细设计

#### 1. 接入层
*   **API 网关**：负责处理 HTTPS 卸载、IP 黑白名单、粗粒度的并发限流（防 DDoS / CC 攻击）。
*   **统一 API 规范**：对外暴露完全兼容 OpenAI 标准的接口（如 `/v1/chat/completions`, `/v1/images/generations`），让下游客户端无缝切换。

#### 2. 业务逻辑层
这是整个系统的心脏，主要包含以下几个核心引擎：

*   **鉴权与配额模块**：
    *   校验客户端携带的 `Access Token`。
    *   读取该 Token 对应用户的可用模型权限和剩余额度。
*   **渠道路由与分发模块**：
    *   **模型映射**：例如用户请求 `gpt-4o`，系统查找哪些可用渠道配置了该模型。
    *   **负载均衡**：按照权重或优先级（Priority）选择一个具体的上游渠道（如 Azure-EastUS）。
    *   **容错与重试机制**：如果请求超时或返回错误（如 429 限流、500 内部错误），自动屏蔽该渠道一段时间，并自动**无缝重试**下一个渠道。
*   **统一格式转换模块**：
    *   将不同的 API 格式互相转换。例如：底层调用的是 Claude，但对外暴露的是 OpenAI 格式，该模块负责将 Prompt、System Message、响应结果进行双向转换。
*   **计费引擎**：
    *   **流式计费**：对于 LLM 的 SSE（Server-Sent Events）流式输出，边生成边计算 Token 数量。
    *   **预扣费与回退**：请求前预扣减预估额度，请求结束后计算实际花费，多退少补。

#### 3. 数据与存储层
*   **关系型数据库**：存储用户、Token、渠道配置、兑换码、计费规则、日志总账。
*   **Redis 缓存**：
    *   缓存用户 Token、配额信息，减轻数据库读压力。
    *   渠道健康状态缓存（记录某渠道是否处于临时熔断状态）。
    *   分布式锁（防止高并发下的额度超额扣减）。

#### 4. 异步任务层
对于耗时较长的非对话类 API（如 Midjourney 画图、Suno 音乐生成、长文本向量化）：
*   请求接收后，生成一个 `Task ID`，丢进消息队列。
*   Worker 节点轮询上游 API 状态，完成后通过 Webhook 或 WebSocket 通知客户端。

### 三、 核心工作流：一次 Chat 请求的生命周期

1.  **拦截请求**：客户端携带 `sk-xxxx` 请求 `/v1/chat/completions`。
2.  **身份验证**：从 Redis 中根据 `sk-xxxx` 查出用户，校验状态和可用模型。若无权限或欠费，直接返回 403/429。
3.  **渠道选择**：
    *   筛选支持目标模型（如 `gpt-4`）的渠道列表。
    *   过滤掉当前处于熔断状态的渠道。
    *   按权重随机挑选一个渠道（如渠道A：权重3，渠道B：权重1）。
4.  **额度预扣**：在 Redis 中对该用户的额度进行预扣减（乐观锁/分布式锁）。
5.  **请求转发与转换**：
    *   将请求体按目标渠道的规范进行修改（如修改 Header，替换 API Key）。
    *   如果是流式，建立缓冲池开始向下转发。
6.  **结果统计与扣费**：
    *   请求完成后，获取上下游的 Token 消耗量。
    *   计算实际费用（`输入Token * 输入单价 + 输出Token * 输出单价`）。
    *   回退预扣费，扣除实际费用，将真实消费数据写入数据库。
7.  **日志记录**：将请求耗时、Token数、目标渠道、是否成功等异步写入日志表。

### 四、 关键技术难点与解决方案

#### 1. 高并发下的 Stream 转发
*   **难点**：大模型的流式输出连接通常持续很久（几十秒到几分钟），如果并发高，会耗尽服务器的 TCP 连接或线程资源。
*   **解决方案**：后端语言建议选用原生支持高并发的语言（如 **Go** 的 Goroutine，或 **Node.js** 的异步非阻塞 IO）。确保 Gateway 不缓冲流数据（Nginx 需关闭 `proxy_buffering`）。

#### 2. 渠道健康检查与熔断机制
*   **难点**：上游 API 不稳定，经常遇到余额用尽、封号、网络波动。
*   **解决方案**：
    *   **自动禁用**：当某渠道连续返回 401 (Unauthorized) 或 429 (Rate Limit) N 次后，自动将其标记为禁用状态。
    *   **无缝重试**：如果上游超时或返回 5xx，系统在后台默默将请求重定向到下一个备用渠道，对客户端完全透明。

#### 3. 防止额度透支
*   **难点**：高并发下，多个请求同时到达，读取到同一个用户的剩余额度，导致最后扣费时额度变成负数。
*   **解决方案**：在请求开始前，使用 Redis 执行 **DECRBY** 原子操作进行预扣费。如果结果 < 0，则拒绝请求并回滚。请求结束后再根据真实消耗修正数据。

#### 4. 并发限制
*   **难点**：不同渠道、不同用户对并发（RPM/TPM）限制不同。
*   **解决方案**：基于 Redis 实现分布式令牌桶算法或滑动窗口算法，对单用户或单渠道实施严格的并发控制，避免触发上游封号。

### 五、 推荐技术栈

*   **后端开发语言**：**Go**（首选，性能高，并发处理极强，New API 也是基于 Go 构建的） / Java / Node.js。
*   **Web 框架**：Gin (Go) / Spring Boot (Java) / Express (Node)。
*   **前端控制台**：React + Ant Design Pro / Vue + Element Plus。
*   **数据库**：PostgreSQL（强烈推荐，适合复杂的计费与账单查询） / MySQL。
*   **缓存**：Redis (Cluster模式)。
*   **部署**：Docker + Docker Compose（适合中小规模），Kubernetes（适合大规模高可用）。
